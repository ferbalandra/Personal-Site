So, What is a Design System?

Style guide or visual pattern library, Design tooling, Component library, Code usage guidelines and documentation, Design usage documentation, Animation language, Voice and tone guidelines

Style guide or visual pattern library a visual group of all the components that encompass a brand. Design tooling is a component broken down in one place and you can see all of them as standalone items.

Component library is the code and interactions attached to each element, how to use them and where. Code usage guidelines and documentation is a guide on how to write the code the usage of the elements where the design team can indicate the purpose of an item and make decisions of usage gidelines. Carbon IBM has a really good example of this design usage documentation.

Animation language is a standarization of some interactions that can be drawing from actual items that the brand makes or uses like IBM uses examples to use as animations

Voice and tone guide, MailChimp has a good voice style guide voice is what you are saying but the tone is how you say it. See their voice and tone Guide Voice can also be added to error messages

Reasons why Design systems fail

Define what to means of a design system to fail: Nobody Uses it. If noone uses it then the benefit of it get lost.The goal is to create Happy Design Systems.

  • Scale Good Standards
  • Build-in Accessibility
  • Unify Component Style
  • Unify Component Code
  • Reduce Code Cruft
  • Speed Up Product Delivery

How to build systmes that work

Investment To have a successful design system, you need to make a continuous effort to invest resources into it. A successful design system is like a successful exercise program. It needs to be consistent and continuisly working on it and kept up to date that adapts to changes and new needs. Designs systems becasuse they scale they can get really large.

Comunication should include as many people as possible, communication is multidirectional. It is important to ask users what they want, listen to them and make user feel like part of the process

Buy-in Showing is more powerful than telling. Creating a small example to show is a great way to find bugs, incorporate feedback and get buy in. A hackathon is a great way to show examples that are quick.

Solid ArchitectureBuild a system that can grow and remove parts, systems do better when they can change and evolve easier. A good example is where there is a system that builds from a core and adds different modules to it. Core like variables, colors, sizing, fonts etc and they build up modular. Another important aspect is to creatre a semantic Versioning where they are Major. Minor. or Patch versions. Having a system to name things is very useful and keep them standirized between design, naming values on code and verions togehter.

Namespace since the begining where you tie classes and components, where you also create a system of naming elements and everyone uses the same
